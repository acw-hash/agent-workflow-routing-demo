Callouts: 

Agents and Workflows: Agents are declared in *.agent file and workflows are declared in *.fdl file 

Workflow: Workflow has .fdl file.  

It has nodes, which are the agents themselves. By default, the agents are referred to by the nodes[].id which is the agent name. However, if multiple agents with same name exists (which would happen if deploy was run multiple times), it is better to specify nodes[].agent.id to point to agent id, which is guaranteed to be unique at all times.  

It has orchestration, which is how the control should flow between agents using events. Here is where input and output flow from one agent to another is specified as well.  

There are 2 names for how data is stored. The term variables are only for workflows. Agent ‘has’ output and workflow ‘has’ variables. 

In orchestration,  

there are Listen_for which can be thought of as event-listener declarations. These events can be standard system defined or it can be agent defined events.  

Each event-listener declaration can be one of multiple types. Type can be update  or emit or node_invocation 

Update when agent output needs to be updated to workflow variable.  

Emit when an event is to fired. 

node_invocation when another node/ agent should be invoked. 

 

Samples:  

 

Workflow playground: We create agent and workflows in vcode; connect to Foundry project; deploy both agents and workflow to Foundry. Agents can be found in the Foundry project. The workflows will not visible there. Correspondingly, there is no playground for workflow in Foundry. The vscode extension allows for workflow playground in vscode itself.  

Deploy agent and workflow: There is a file called deploy.sh that has the logic on deploying both agents as well as workflows.  

Deploy.sh works such that it uses the file extension to discriminate between agent and workflow. If the ID field in the agent or workflow file is not specified, it creates, else updates the existing one locating by specified id.  

It’s good to prevent multiple copies of same agent getting deployed as agent deletion in Foundry is tedious (one by one) and workflow deletion is not possible.  

Connect Workflow to Foundry: Clicking on connect with the new SDK has the widget to connect to the subscription and the Azure foundry project.  

Workflow Inference:  

Workflow cannot be invoked from Foundry UI or any other SDK. These can be invoked using REST API calls. Work in progress.
